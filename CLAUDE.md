# AI Home Profile — Claude Code Context

## What This App Is

AI Home Profile is an Android app that helps homeowners maintain their property. The user completes a one-time home profile (property type, age, construction, heating, roof, outdoor features, safety systems) and receives a personalised 12-month maintenance calendar generated by the Anthropic API. A WorkManager background agent sends seasonal push notifications at the right time of year. Users log completed tasks, building a maintenance history over time.

**The core value proposition:** every task feels specific to *this* home, not a generic checklist. A 1960s solid-wall house with cast iron gutters and a 15-year-old gas boiler has almost nothing in common with a 2015 new-build — the AI makes that distinction real.

## Architecture Rules

- **MVVM strictly enforced.** ViewModels own UI state and expose it as `StateFlow`. Screens observe state — they never call repositories directly.
- **Repository is the single source of truth.** All data access goes through repositories. ViewModels call use cases; use cases call repositories.
- **Domain layer has zero Android imports.** Use cases are plain Kotlin classes — no Context, no Android framework dependencies.
- **Single-activity app.** All navigation via Compose Navigation with a single `NavHost`.

## Database — Room Tables

### HomeProfile
Single row. Core home details collected during onboarding.
- `property_type`, `year_built_range`, `wall_type`, `roof_type`
- `heating_type`, `boiler_type`, `boiler_age_range`, `last_boiler_service`
- `has_garden`, `gutter_type`, `has_smoke_alarms`, `has_solar`
- `location_climate_zone` — derived from postcode region, NOT the raw postcode
- `profile_complete: BOOLEAN`

### MaintenanceTask
Generated by AI at profile completion. One row per task.
- `task_key` — unique string from AI e.g. `boiler_service_annual`
- `title`, `category`, `why_this_home` — AI-generated explanation
- `optimal_month_start`, `optimal_month_end` — integers 1–12
- `urgency` — low | medium | high | critical
- `effort_diy_hours`, `difficulty`, `recurrence`
- `is_completed_this_cycle: BOOLEAN`
- `snoozed_until: INTEGER` — Unix timestamp, nullable

### TaskLogEntry
Permanent log of completed tasks. Never deleted.
- `task_id` — FK to MaintenanceTask
- `task_title`, `task_category` — snapshot (task may change)
- `completed_at: INTEGER` — Unix timestamp
- `notes: TEXT` — optional user note
- `amended: BOOLEAN`

### NotificationLog
Audit trail for sent notifications.
- `notification_type` — heads_up | follow_up | urgency | critical
- `task_ids_included` — JSON array
- `sent_at: INTEGER`

## AI Integration

### Calendar Generation
- **When:** once on profile completion, and when a core profile field is edited
- **Model:** `claude-sonnet-4-20250514`
- **Endpoint:** `POST https://api.anthropic.com/v1/messages`
- **Prompt structure:** system prompt (task generation instructions) + home profile block + climate zone + task JSON schema
- **Response:** JSON array of tasks — parse and store to Room
- **Max tokens:** 4000
- **Target latency:** under 8 seconds

### Task JSON Schema (enforce in prompt)
```json
{
  "task_key": "string",
  "title": "string",
  "category": "heating|roofing|plumbing|garden|safety|structure|electrical|general",
  "why_this_home": "string — 2-3 sentences specific to this property",
  "optimal_month_start": "integer 1-12",
  "optimal_month_end": "integer 1-12",
  "urgency": "low|medium|high|critical",
  "effort_diy_hours": "number|null",
  "difficulty": "diy_easy|diy_moderate|professional_recommended",
  "recurrence": "annual|biannual|quarterly|one_off|as_needed",
  "applies_because": "string — audit tag e.g. gas_boiler or pre_1980_construction"
}
```

### Critical AI Rule
The AI must only generate tasks that apply to the specific home profile. If a field is null or unknown, do not generate tasks that depend on it. The `applies_because` tag exists to make this auditable.

## Privacy Rules — Non-Negotiable

These rules must be enforced in code, not just policy:

1. **Raw postcode is never stored.** Derive a climate zone string (e.g. `northern_uk_high_rainfall`) and store only that.
2. **No analytics SDK.** No Firebase Analytics, no Mixpanel, no third-party event tracking in Phase 1.
3. **No cloud storage in Phase 1.** All Room data stays on device. No backup to any remote server.
4. **AI API payload must be reviewed** in `CalendarGenerationUseCase` before every call — it must contain only: structured home profile fields, climate zone, and task schema. No free-text user input that could contain PII.
5. **No personal identifiers in logs.** Debug logs must not contain postcode, address, or any home-identifying string.

## Seasonal Nudge Worker

`SeasonalNudgeWorker` runs weekly via WorkManager (requires network: false, requires charging: false).

### Trigger Logic
| Condition | Notification Type |
|---|---|
| Task window opens within 14 days | `heads_up` |
| Task in window, incomplete after 21 days | `follow_up` |
| Task window closes in 7 days, incomplete | `urgency` |
| Critical task overdue | `critical` |

### Notification Rules
- **Maximum 1 notification per week.** Multiple triggers are batched into a single summary notification.
- **Quiet hours:** 9am–8pm local time only.
- **Critical tasks** (smoke alarms, CO detectors, gas safety) bypass the weekly limit.
- Notification copy is **pre-written per trigger type** — no AI call at notification time.
- Log every sent notification to `NotificationLog`.

### Battery-Saver Device Handling
Xiaomi, Huawei, and some Samsung devices aggressively kill WorkManager. On first notification delivery, detect the device manufacturer and prompt the user to whitelist the app in battery settings if needed.

## Onboarding Flow

5 profile sections collected across screens O1–O8:

1. **Property Basics** — type, ownership, year built, bedrooms, floors, location
2. **Construction** — wall type, insulation, roof type, loft insulation, glazing, basement, conservatory
3. **Heating & Hot Water** — heating type, boiler type/age/last service, hot water tank, radiators, underfloor
4. **Outdoor & Garden** — garden, driveway, gutters, fascias, boundary, flat roof sections
5. **Systems & Appliances** — smoke/CO alarms, stop valve, septic tank, solar, EV charger, security

All inputs are **tap-to-select only** — no free text except optional notes fields. Every screen has a 'Skip this section' option. Unknown/skipped fields are handled gracefully by the AI (tasks requiring that data are simply not generated).

### Calendar Regeneration
Triggered only when a **core field** changes: heating type, boiler type, roof type, wall type, or construction era. Warn the user before regenerating. Minor field changes (e.g. garden type) do not trigger regeneration.

## Coding Conventions

- **Compose functions:** PascalCase, annotated with `@Composable`
- **ViewModels:** suffix `ViewModel`, one per screen or logical group
- **Use cases:** suffix `UseCase`, single public `invoke()` function
- **Repository interfaces:** in domain layer; implementations in data layer
- **Constants:** in a top-level `Constants.kt` — no magic strings in business logic
- **Coroutines:** `viewModelScope` in ViewModels; `IO` dispatcher in repositories
- **Error handling:** sealed `Result` class — never expose raw exceptions to UI layer
